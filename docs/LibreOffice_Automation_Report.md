# Final Report: The Journey to Robust Local Office Automation

This report chronicles our investigation into the **WaterPistolAI/libreoffice-mcp** project, the technical hurdles overcome during testing in a containerized environment (Flatpak/CachyOS), and the final blueprint for a production-ready Docker deployment.

---

## 1. The Quest: Bridging AI and Local Documents
The goal was to evaluate a system that allows an AI agent (like **Hermes**) to not just *read* but actively **edit and manage** local office files (Excel, Word, PowerPoint) without relying on fragile, unmaintained scripts or expensive, privacy-compromising cloud APIs (Microsoft Graph, Google Drive).

### The "OooDev" Breakthrough
Our research confirmed that the core of this project’s success is the use of **OooDev**. 
*   **Traditional Method:** Raw PyUNO (LibreOffice's native bridge) is archaic, poorly documented, and prone to silent crashes.
*   **The Project's Method:** OooDev provides a modern, Pythonic wrapper. It transforms complex internal C++ interfaces into simple commands like `doc.save_doc()` or `sheet.set_val()`.

---

## 2. The Journey: Overcoming Sandbox Challenges
Testing this on **CachyOS** via **Flatpak** presented a unique "worst-case scenario" for automation—perfect for identifying potential points of failure.

### Technical Hurdles & Solutions:
1.  **The Runtime Disconnect:** The host's Python (3.14) could not see the Flatpak's internal LibreOffice libraries.
    *   *Solution:* We manually mapped the `URE_BOOTSTRAP` environment variable to the Flatpak's internal `fundamentalrc` file and injected `/app/libreoffice/program` into the `PYTHONPATH`.
2.  **The Bridge Magic:** LibreOffice uses a custom import hook for its `com.sun.star` modules. 
    *   *Solution:* We manually registered `uno._uno_import` into Python's `builtins.__import__` to enable the "magic" imports required for automation.
3.  **The Memory Ceiling:** While simple spreadsheets and 20MB Word documents were edited successfully, a 27MB PowerPoint file triggered a `Binary URP bridge disposed` error.
    *   *Finding:* Large files (especially presentations) require significant shared memory and RAM, setting a clear performance boundary for headless instances.

---

## 4. Final Blueprint: Best Practices for Production (Docker)
To avoid the complexities of Flatpak and version mismatches, a **Dockerized deployment** is the gold standard.

### Core Strategy:
*   **Base Image:** Use `python:3.11-slim-bookworm`. Debian's `glibc` is required for the UNO bridge; avoid Alpine.
*   **Process Management:** Always use `tini` as an init system. This prevents LibreOffice from leaving "zombie" processes that eventually crash the container.
*   **Memory Allocation:** Set `--shm-size=2gb` and ensure the container has at least 2GB of RAM to prevent bridge disposal on large files.
*   **Font Fidelity:** Bind-mount your host fonts (`/usr/share/fonts`) to ensure PDFs and previews generated by Hermes look professional.

The **WaterPistolAI/libreoffice-mcp** approach transforms LibreOffice from a desktop application into a powerful **Agentic Toolset**.

This allows for a seamless workflow where Hermes summarizes an audit report, updates a student roster, and saves it directly to your Dropbox—all without a single byte of data leaving your server.
